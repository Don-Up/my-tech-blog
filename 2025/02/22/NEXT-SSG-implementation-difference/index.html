The Evolution of Static Site Generation (SSG) in Next.js

2025-02-22
#SSG
Next.js

Introduction

Next.js has evolved significantly over the years, and one of the major changes introduced in Next.js 13 was the shift in how Static Site Generation (SSG) is implemented. Prior to version 13, SSG relied on a traditional model using getStaticProps and getStaticPaths. However, with the introduction of the App Router and React Server Components (RSC) in Next.js 13, the approach to SSG changed fundamentally.

In this blog, we’ll explore how SSG worked before Next.js 13 and how it has changed with the new App Router paradigm.

SSG Before Next.js 13 (Using getStaticProps)

Before Next.js 13, SSG was implemented using getStaticProps and getStaticPaths, which were functions specifically designed to fetch data at build time.

How It Worked

  1. getStaticProps was used to fetch data at build time.
  2. getStaticPaths (when used with dynamic routes) was used to predefine paths for static generation.
  3. The pages generated were completely static and served via the CDN.

Example: SSG in Next.js 12

Key Characteristics

  • Build-time execution: Data is fetched at build time and baked into the HTML.
  • No runtime fetching: Since the HTML is static, there is no need for client-side fetching unless explicitly done.
  • Faster page loads: Pre-rendered content is served from the CDN, making it very fast.
  • Limited flexibility: Any new content requires a full rebuild or revalidation using ISR (Incremental Static Regeneration).

SSG After Next.js 13 (Using the New App Router)

With Next.js 13, the introduction of the App Router (app/ directory) and React Server Components (RSC) changed how SSG works. Instead of using getStaticProps, static data fetching is now done directly inside Server Components.

How It Works Now

  1. Server Components by default: In Next.js 13+, all components inside app/ are server components by default, meaning they can fetch data without useEffect or getStaticProps.
  2. Fetching inside async components: Instead of defining a separate function like getStaticProps, you can now fetch data directly inside a server component.
  3. Automatic static optimization: If no dynamic data (e.g., user authentication) is present, Next.js automatically pre-renders the page as static at build time.

Example: SSG in Next.js 13+

Key Characteristics

✅ No getStaticProps or getStaticPaths: Simply fetch data inside the component.
✅ Server Components optimize rendering: Data fetching happens on the server, reducing client-side JavaScript.
✅ Automatic static generation: If no dynamic content is detected, Next.js automatically statically generates the page.
✅ Flexible data fetching: If fetch uses { cache: 'no-store' }, it will be treated as dynamic (SSR instead of SSG).

📊 Major Differences Between Before and After Next.js 13

Feature Before Next.js 13 (Pages Router) After Next.js 13 (App Router)
SSG Method getStaticProps Fetch data inside async components
Dynamic Path Handling getStaticPaths params from page.jsx props
Server Component Support ❌ No ✅ Yes (by default)
Automatic Static Optimization ✅ Yes ✅ Yes (implicit)
Revalidation (ISR) revalidate in getStaticProps { next: { revalidate: X } } in fetch

🤔 Which Approach Should You Use?

  • If you’re migrating from Next.js 12, you should start using the App Router and move away from getStaticProps in favor of fetching inside Server Components.
  • If you’re using static content that rarely changes, the new App Router approach is much simpler and more efficient.
  • If you need dynamic content updates, you can use ISR (revalidate in fetch) or Server Actions based on requirements.

🎯 Conclusion

Next.js 13 introduced a simpler and more powerful way to implement Static Site Generation (SSG). The introduction of Server Components and the App Router has eliminated the need for getStaticProps and getStaticPaths, making data fetching more intuitive. This shift reduces boilerplate, improves performance, and increases flexibility.

If you’re still using the old Pages Router, now is a great time to migrate and take advantage of these modern Next.js features!
